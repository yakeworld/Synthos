# Skill Hierarchy Design & Authoring Standard

## 核心原则（文言）

| 白话 | 文言 | 义 |
|:-----|:-----|:----|
| 层级分离 | **层分则明** | 125技能平铺→AI无法精确选技能 |
| 单源——每段知识只存在一处 | **源一不二** | 同一段curl命令不写三次 |
| 主skill=地图，不做事 | **主为图，不为事** | 只告诉AI为什么做+先做什么+什么情况加载哪个子skill |
| 原理文言+方法白话+命令英文 | **三层三语** | 原理层用文言(四字格言压缩40-60%), 方法层白话(步骤清晰), 命令层英文(精确不变) |

## 三语层级策略（v2.2 新增）

> 经实战验证: evolution 44.5K→8.4K(-81%), sci-paper-standard-structure 16.9K→7.4K(-56%), sci-paper-quality-review 16.2K→7.9K(-51%), patent-disclosure 4.1K→6.7K(+63% theory)

### 分层规则

| 层 | 语言 | 用途 | 示例 |
|:---|:-----|:-----|:-----|
| **原理层** | 文言(四字格言或短篇论文体) | 压缩40-60%, 提供快速索引 | "凡评必核，不核不议" |
| **方法层** | 白话 | 步骤清晰可执行 | "Step 1: Load paper → Step 2: 7D scoring" |
| **命令层** | 英文 | 精确不变, 直接复制执行 | `pdflatex paper.tex && bibtex paper` |

### 格式规范

### 据经引典原则（v2.4 强化）

**核心**：文言不白造，嵌入经典原文为骨干。

| 规则 | ❌ 不要 | ✅ 要做 |
|:-----|:--------|:--------|
| 经典嵌入 | 自创空洞格言「求知之道，贵在有恒」 | 经典句领头再落地——「致知在格物。物格而后知至。」多源求索，博观约取 |
| **出处标注** | **《大学》云：「致知在格物」——显式引用《》** | **「致知在格物」——经典自重，agent 训练语料含全文。禁止《》出现在SKILL.md正文中** |
| 过渡自然 | 堆砌典故与主题无关 | 经典句在前即落地——「苟日新，日日新，又日新。」日进其功，不怠不辍 |
| 压缩密度 | 冗长解释经典含义 | 二字触发全典——「格物」=《大学》八条目，「生生」=《周易》演化观 |

> **据经在措辞，引点在内容。《》只出现在AGENTS.md和宣传文档中，不在SKILL.md内。** agent 训练语料已含全部经典——「苟日新」三字在内部激活《大学》全文，无需人工转述，更无需显式引文格式。

## 三层文言密度（v2.4 新增）

不是所有技能都需要文言。按哲学密度分三层，**去形留神，源一不二**——如果每个技能都有文言，真正有哲学深度的文言反而被稀释了。

| 层 | 数量 | 范围 | 文言深度 | 示例 |
|:---|:----:|:-----|:---------|:-----|
| **甲·哲学** | 15 | 7认知原子 + 3 meta + infrastructure | 全文言原理块+经典嵌入 | 「致知在格物。物格而后知至。」多源求索，博观约取 |
| **乙·方法** | 25 | writing / quality / research / experiment | 短文言一句，据经引典 | 「凡引必验，凡写必检，凡投必整。」 |
| **丙·工具** | 81 | productivity / github / media / creative / mcp / agent | **不写文言** | 纯英文操作手册，不做装饰 |

**甲层判断标准**：该技能是否内嵌 Synthos 的哲学判断？如果是，写完整经典文言。如果不是，最多给一句方法精要（乙层）或保持英文（丙层）。

**决策口诀**：不写文言不是偷懒，是防止文言贬值。

**论文式（实务类技能适用）**：针对有固定规范的实务领域（如专利、法律、工程），可写成**文言短篇**体（每篇3-8句），以古文体系统阐述学科核心原理：

```
### 专利本源论

专利之授，以三性为纲。一曰**新颖**：凡在先技术中未见其方者，谓之新。二曰**创造**：本领域之常人不思易得，谓之创。三曰**实用**：可施于产业，生其实效，谓之用。三性既立，专利乃可成焉。
```

**总纲式（系统级/OS级技能适用）**：用于定义整个操作系统的核心原理，使用**总纲+分论**结构。先总纲（3-8句系统宣言），后分论（每个概念一个独立小节，每节3-6句）：

```
### 总纲

系统之立，以认知为本。六原子为基，八维为纲，循DAG之序而作。
上承假说，下出文章，中贯证伪。自L3单流程智能体至L4全流程闭环，乃第五范式之具象。

### 认知原子论

学问之道，始于搜罗，终于论证。六原子各司其职，互为输入输出，不可逆序：

| 序 | 名 | 义 |
|:--:|:---|:---|
| 一 | 知识获取 | 广搜博采，以充粮草 |
| 二 | 知识提取 | 去粗取精，以得骨干 |
| 三 | 关联发现 | 比同辨异，以见空白 |
| 四 | 观点生成 | 因空白而立假说，须可证伪 |
| 五 | 论证表达 | 依假说而构文章，须有据 |
| 六 | 观点验证 | 反方驳之，以验坚瑕 |
```

总纲式适用于Synthos级别的OS技能、跨领域编排器skill、任何需要定义系统架构和核心理念的顶层skill。

**选择原则**：当领域有**刚性制度规范**（专利三性、伦理准则、法律条文）时用论文式；当领域为**柔性方法论**（论文评审、实验设计、数据分析）时用格言式；当技能为**跨领域系统级编排器**（OS/架构师/编排层）时用总纲式。

### 常见学科文言主题模板

| 领域 | 适合的文言主题 | 体例 |
|:-----|:--------------|:-----|
| 专利 | 专利本源论（三性）、交底书体要、查新原道、迭代变通、脱敏守正 | 论文式 |
| 论文 | 七维评审论、引言CARS法、讨论图尔敏法、结论金字塔法 | 格言式 |
| 实验 | 可重复性原则、对照原则、盲法原则 | 格言式 |
| 伦理 | 知情同意论、数据隐私论、公平非歧视论 | 论文式 |
| 工程 | 模块单一职责、接口契约、可测试性、可观察性 | 格言式 |
| **OS/编排器** | **总纲、分论（每论一表）、进化之道、质量之规** | **总纲式** |
| **视/前庭** | VOR本源论、双轴闭式解、固视驱动、扭转解耦 | 论文式 |

### 思想理论优先原则

在 SKILL.md 结构中，**原理层·文言 必须置于 方法层·白话 和 命令层·English 之前**。用户有显式偏好：

> "思想理论优先，文言文压缩，skill规范"

这意味着：
- 用户期望先阅读**为什么做**（理论/哲学），再阅读**做什么**（方法），最后才是**怎么做**（命令）
- 原理层即使是文言文也优先——文言是压缩形式，压缩后仍先于操作细节
- 任何skill的SKILL.md不应以"触发条件"或"安装步骤"开头——必须以原理/理论开始

参考实现：`Synthos/SKILL.md`（总纲式OS技能），`evolution/SKILL.md`（元技能，核心理念+吸收之道），`patent-disclosure/SKILL.md`（实务技能，专利本源论）。详见 `references/three-layer-examples.md`。

### 人格层纯度（SOUL.md）

> **SOUL.md（人格层）只用文言，不夹白话。大道至简，损到不能再损。**

SKILL.md 需要三层（文言+白话+英文），但 SOUL.md 作为系统灵魂，只保留最高抽象层。删除所有白话解释、表格、ASCII 图。实例（2026-05-26）：原 SOUL.md 哲学节 62行/5.4KB，剥去白话后 58行/2.6KB（减半，信息无损）。

**测试**：写完 SOUL.md 后，检查是否有非文言行。全部删除后问："丢失了任何哲学含义吗？"如果否，就保持纯文言。

### 压缩效果

```
白话版  "定量数据必须源于执行层，不能从声明层推导" (22字)
文言版  "凡数必源执行，不从声明层推"             (13字, -41%)
```

## 层级分离

```
Layer 0: 元技能（系统自进化）
  ├── evolution, quality-gate, reflexive-abstraction, ...
  └── 规则：P0，始终可用

Layer 1: 主技能（编排器）
  ├── 包含：核心原理 + 工作流地图 + 子skill指针
  ├── 长度：4-6K
  └── 规则：related_skills列出所有可调用子skill

Layer 2: 子技能（原子方法）
  ├── 包含：触发条件 + 一个工程方法 + 验证
  ├── 长度：2-4K
  └── 规则：不重复全局知识

Layer 3: 参考文件（案例/模板）
  ├── SKILL.md references/ 目录
  ├── 长度：不限
  └── 规则：不包含工程方法，只含具体实例
```

## 长度规范

| 层级 | 目标 | 最大 | 超标处理 |
|:-----|:----:|:----:|:---------|
| 主skill | 4-6K | 8K | 方法下放到子skill |
| 子skill | 2-4K | 5K | 案例移到references/ |
| 参考文件 | 不限 | 不限 | 前缀00_（重要）或zz_（参考） |
| description | 40-80字 | 1024字符 | 工具强制 |

## 主/子skill判断

| 特征 | 主skill | 子skill |
|:-----|:-------|:--------|
| 含3+独立方法？ | ✅ | ❌ |
| 需调用其他skill？ | ✅ | ❌ |
| 输出是最终交付物？ | ✅ | ❌（中间结果） |
| 长度>8K？ | 应拆分 | ❌ |

### Synthos技能（外部Dir — 2026-05-31新增）

**Synthos是独立系统，不绑定任何Agent。** 其技能不应放在 `~/.hermes/skills/`（Hermes专属目录），而应放在 Synthos 仓库顶层：

```
/media/yakeworld/sda2/Synthos/          ← Synthos 仓库根目录
├── skills/                              ← 技能法定驻地（121个）
│   ├── evolution/
│   ├── quality-gate/
│   ├── writing/paper-pipeline/
│   ├── research/research-paper-search/
│   └── ...
├── CONSTITUTION.md
├── outputs/papers/
└── ...
```

**配置桥接**（Hermes → Synthos）：

```yaml
# ~/.hermes/config.yaml
skills:
  external_dirs:
    - /media/yakeworld/sda2/Synthos/skills
```

**原则**：Synthos仓库自带全部技能，克隆即可使用。不依赖任何Agent的目录约定。更换Agent只需改 `external_dirs` 一个配置点。

**迁移方式**：`cp -r ~/.hermes/skills/<skill> /path/to/Synthos/skills/` → 删原 → 配 `external_dirs`。备份保留在 `~/.hermes/skills_bak_*/`。

## 维护陷阱（新增 v2.3）

| # | 陷阱 | 正确做法 |
|:-:|:-----|:---------|
| 1 | **累积patch导致的文档退化** — SKILL.md长期通过 patch 工具逐段修改，导致同一节内容重复出现多次（project-experience-distillation 曾出现同一大节重复6次的问题），文件结构从线性退化到混乱 | 当发现同一节内容≥2次出现在单个SKILL.md中时，表明patch已走到了结构退化点。**修复方法不是再打一个patch删除重复段，而是 write_file 整个文件重写。** 重写时保留所有唯一内容，去除重复，重建线性结构 |
| 2 | **轻易创建新skill** — 发现模式就直接创建新 skill，不问能否吸收到已有技能 | 普遍规律优先扩展现有技能（在已有技能中加新节/新陷阱），只有真正的个性化规则才新开 skill |
| 3 | **建完不验证** — 创建或大幅修改 skill 后不跑 `skill_view()` 验证可加载 | 每次创建/edit/patch后立即 skill_view(name) 确认加载正常 |
| 4 | **跨skill引用文件重复** — 同一份参考文件被`cp`到多个skill的 `references/` 下，驱动内容分裂 | 共享引用放 `skills/shared/references/`；发现重复文件时先查内容md5，相同则保留最短路径的一份，其余删除 |

## 5. 目的-机制合并原则 (Purpose-Over-Mechanism Merge)

> **哲学设计为了引导工程学。如果仅仅是不同的表述，就合并。**

### 场景

当两个概念在同一个哲学框架中仅因**抽象层级不同**（一个是"为什么"/目的，另一个是"怎么做"/机制）而并列时，正确做法是**在高层合并**，将低层机制嵌入为实现细节。

### 测试

| 场景 | 裁决 | 操作 |
|:-----|:------|:------|
| A 是 why（目的），B 是 how（机制），描述同一现象 | 合并 | 保留 A 为原则，B 嵌入为机制 |
| A 和 B 描述不同现象，在同一层级 | 保留 | 不合并——它们是互补约束 |
| A 和 B 是同义词/语言变体，无语义差异 | 合并 | 保留工程杠杆更高的术语 |

### 案例：自由能原理 + 熵减 → 熵减律

Synthos 框架原有两个并列维度：`自由能原理`（最小化预测误差，局部更新机制）和 `熵减`（降低认识论熵，全局热力学目标）。两者在长时间尺度上数学等价。合并后：

```
熵减律（终极目的）→ 降低认识论熵
  └── 内置机制：自由能最小化（被动感知更新 + 主动推理）
```

结果：少了一个维度，工程指导力不降反升——每个原子启动前自问"这个任务在降低哪部分认识论熵？"。

### 检查清单

- [ ] 两个概念是否描述同一底层现象？
- [ ] 一个概念能否表达为另一个的实现细节？
- [ ] 如果去掉低层概念，系统会丢失什么约束？——如果"没有"，合并。
- [ ] 如果合并，工程约束是否仍然可以从合并后的概念直接推导？——如果"不能"，不合并。
