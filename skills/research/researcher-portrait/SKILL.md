# Researcher Portrait — 研究者学术面貌档案

> 创建一份可信任、可溯源、有据可查的中国研究者面貌档案。

## 核心原则

| 原则 | 说明 |
|:-----|:------|
| **数据可溯源** | 每条数据标注来源（网站/数据库/用户确认），不编造 |
| **状态透明** | 明确标注"已发表"vs"已完成"vs"拟投稿"，不混用 |
| **错漏可修** | 建立用户纠错闭环——被纠正后即刻全量修正，不道歉只修复 |
| **交叉验证** | 网站数据 → NotebookLM分析 → 用户确认，三层校验 |
| **来源标注** | 商业计划书等衍生文档中，每项数据标注置信度标签 |

## 标准章节结构（11章）

```
研究者面貌_姓名.md
├── 一、身份与职务（职称/导师/人才称号/学术兼职/教育经历）
├── 二、科研项目（基金号/来源/金额/角色/时间）
├── 三、已发表论文（SCI+中文核心，分第一/通讯作者与非第一作者）
├── 四、专利成果（发明已授权/发明实审中/实用新型/软著）
├── 五、已转化产品（量产/原型机/推广应用/软著）
├── 六、继续教育项目（国家级/省级/市级）
├── 七、平台资源（实验室/中心/病区）
├── 八、临床数据（门诊量/检查量/成功率）
├── 九、系统/方法论开发（如自研AI科研系统、方法论框架等）
├── 十、研究聚焦领域（按主次排列，每领域附支撑数据）
└── 十一、待补充信息（NotebookLM分析建议补充的板块）
```

### 第九章说明：系统/方法论开发

当研究者开发了自主系统（如Synthos科研操作系统）、方法论框架（如CRISP-DM Helix）、或工具链时，应单独设立此章。包含：系统定位、规模（论文产出/技能数量/质量体系）、核心组件、与临床/基础研究的关系。这在商业计划书、人才申报等场景中能显著增强"方法论创新"维度。

## 典型应用场景

| 场景 | 用途 | 重点章节 |
|:-----|:------|:---------|
| **商业计划书** | 团队介绍/技术来源 | 三(论文)/四(专利)/五(产品)/九(系统) |
| **课题申报** | 研究基础/成果积累 | 二(项目)/三(论文)/六(继教) |
| **人才评选** | 个人履历 | 一(职务)/三(论文)/四(专利)/十一(待补) |
| **竞赛材料** | 团队实力展示 | 一(身份)+三+四+五+九 |

## Step 1: 确定数据源

| 来源类型 | 特点 | 推荐度 |
|:---------|:-----|:------:|
| **个人/实验室官网**（如WordPress站点） | 最全，含论文/专利/项目/产品 | ⭐⭐⭐ |
| **医院/学校官方简介页** | 身份/职务权威 | ⭐⭐⭐ |
| **PubMed / Semantic Scholar** | 论文清单，缺专利/项目 | ⭐⭐ |
| **Google Scholar** | 论文+引用指标，需翻墙 | ⭐⭐ |
| **CNKI / 万方** | 中文论文，需账号 | ⭐⭐ |
| **用户直接提供** | 最准确，优先使用 | ⭐⭐⭐ |

## Step 2: 全量数据提取

### 2.1 中文网站HTML表格提取（核心技术）

中文学术网站常用HTML表格展示专利/论文数据。**仅用正则匹配ZL/CN号会漏大半**——ZL/CN号正则只匹配到了28行中的9行，漏了19行。必须解析表格结构：

```python
import re, urllib.request

html = urllib.request.urlopen(url, timeout=30).read().decode('utf-8', errors='ignore')

# 定位到专利/论文所在表格区域
section = re.search(r'专利成果.*?软件著作权|论文.*?参考文献', html, re.DOTALL)
if section:
    rows = re.findall(r'<tr[^>]*>(.*?)</tr>', section.group(), re.DOTALL)
    # 🔴 关键：必须用 <tr> 分组，不能只匹配 ZL/CN 号
    for row in rows:
        cells = re.findall(r'<td[^>]*>(.*?)</td>', row, re.DOTALL)
        texts = [re.sub(r'<[^>]+>', '', c).strip() for c in cells]
```

> ⚠️ 2026-05-30 实战教训：3deyes.top的专利表有28行，用ZL/CN正则只匹配到9行。改用`<tr>`标签分组后全部28行都被捕获。

### 2.2 提取的数据类别

| 类别 | 关键字段 | 提取技巧 |
|:-----|:---------|:---------|
| 专利 | 名称/授权号/类型/授权日/排序 | `ZL`开头=已授权 ✅，`CN`开头=实审中 ⏳ |
| 论文 | 标题/期刊/年份/角色(第一/通讯) | 注意区分本人论文vs参考文献——必须明确作者过滤器 |
| 项目 | 名称/基金号/来源/时间/角色 | 注意查重(同一项目可能出现在不同分类表格) |
| 职务 | 机构/职务名称/时间范围 | 按时间线排列 |
| 产品 | 名称/开发阶段/定价 | 区分"量化生产"vs"原型机"vs"推广应用" |

### 2.3 论文发表状态标注（重要陷阱）

**这是最常见的错误来源。不可混用"已完成"和"已发表"。**

| 状态 | 标注方法 | 示例 |
|:-----|:---------|:------|
| 已发表（SCI期刊） | "期刊名, 年份" | "Frontiers in Neurology, 2024" |
| 投稿中 | "期刊名（投稿中）" | "Pattern Recognition（投稿中）" |
| 准备投稿 | "待投稿" | — |
| 内部草稿/质量门通过 | "Synthos内部T1质量通过" | — |

> ⚠️ 2026-05-30 实战教训：商业计划书中"已发表SCI论文5篇"实际所有论文均未发表，被用户当场纠正。此后必须逐篇确认发表状态。**任何模糊表述都是风险。**

### 2.4 专利申请状态判断

| 授权号前缀 | 含义 | 使用标签 |
|:-----------|:-----|:---------|
| `ZL`（如ZL201710628557.3） | 已授权 | ✅ 已授权 |
| `CN`（如CN202210741630.9） | 实审中/申请中 | ⏳ 实审中 |

## Step 3: 数据质量标注

每条数据标注置信度，特别是在商业计划书等衍生文档中使用时：

| 标签 | 含义 | 使用场景 |
|:-----|:-----|:---------|
| ✅ 确凿 | 有明确来源可追溯 | 网站公示数据、用户确认 |
| 🟡 合理估算 | 基于已知数据的推算 | 门诊量、营收预测等 |
| ⚠️ 需核实 | 来源不足或逻辑存疑 | 政策引用、市场价格 |
| ❌ 需修正 | 确认有误的表述 | "已发表"实为"已完成" |

## Step 4: 交叉验证（NotebookLM）

### 4.1 上传文档到NotebookLM

```bash
notebooklm create "研究者面貌分析-{姓名}"
notebooklm use <project_id>
notebooklm source add "$(cat 研究者面貌_姓名.md)" --type text --title "研究者面貌_姓名" --timeout 120
```

### 4.2 多维提问模板（逐轮提问，不合并）

**第一轮：全景扫描**

```bash
notebooklm ask "请全面分析这份研究者面貌文档，从以下维度逐一评审：
1. 完整性：缺少什么关键信息？哪些板块有待补充？
2. 准确性：是否有逻辑矛盾或数据不匹配？
3. 亮点提取：哪些信息最能体现研究者的学术影响力？
4. 格式优化：结构和表达有什么改进空间？
5. 潜在缺失：作为一个完整的研究者profile，可能还缺少什么重要的数据维度？"
```

**第二轮：深层挖掘**

```bash
notebooklm ask "基于前述分析，请具体建议补充以下几项：
1. 该文档缺少的研究聚焦领域内容，请根据已有数据补全
2. 估算H-index和总引用量的大致区间
3. 核心学术网络和长期合作者分析
4. 数据中的重复或归属错误
5. 标题与正文数据量不匹配的修正建议"
```

### 4.3 NotebookLM常见分析发现

| 发现类型 | 典型表现 | 处理 |
|:---------|:---------|:-----|
| 标题低估 | "20+篇"但正文列出37篇 | 改为"40+篇（含非第一作者）" |
| 分类错位 | 教改项目混入科研项目 | 独立设立"教学研究"板块 |
| 数据矛盾 | 专利标注"非第一发明人"但排序=1 | 检查原始数据后修正 |
| 缺奖项/会议 | 无获奖情况和学术交流记录 | 纳入"待补充信息"章节 |
| H-index缺失 | 无任何引用指标 | 标注估算区间+建议独立查新 |
| 发表状态模糊 | "已发表5篇"实为"已完成待投稿" | 逐篇确认后全量修正 |

## Step 5: 用户纠错闭环

**被纠正后的行为**（重要）：
1. 立即查找根因（是数据提取遗漏还是归类错误）
2. 全量重新提取相关数据，**不能只修一条**
3. 修正所有相关文件（profile + 所有衍生文档如商业计划书）
4. **不道歉**，直接更新后报告修正结果："已按{原数据}修正"
5. 将教训写入SKILL.md的陷阱章节

> 实战案例（2026-05-30）：
> - 用户指出专利数不对 → 根因：HTML表格只解析了前9行，漏了后半19行
> - 修正：改用`<tr>`标签分组提取，17项→28项
> - 波及文件：研究者面貌 + 商业计划书（3个章节）+ 质量评估报告
> - 总计4处修正，全部同步完成

## 已知陷阱

| 陷阱 | 表现 | 解决 |
|:-----|:-----|:-----|
| **"已发表"vs"已完成"混淆** | 用户论文未发表却被标注为已发表 | 逐篇确认状态；Synthos论文标注"内部质量通过" |
| **HTML表格解析不全** | 正则匹配ZL/CN号只匹配了1/3 | 使用`<tr>`标签分组，不依赖号码前缀 |
| **专利双报重复计数** | 同一方案在发明+实用新型都出现 | 分别列出但加注，汇总时额外说明 |
| **论文数量标题不一致** | 标题写20篇但正文列了37篇 | 标题用"40+篇（含非第一作者）" |
| **教改项目混入科研项目** | ChatGPT教改被计为科研项目 | 独立设立"教学研究"板块 |
| **专利申请状态遗漏** | 只标了已授权的，漏了实审中的 | 区分ZL(已授权)和CN(实审中) |
| **数据孤立不互引** | 论文/专利/项目数据各自独立 | 建立"研究聚焦"章节串联所有数据 |
| **Synthos系统产出遗漏** | 系统方法论未纳入研究者能力维度 | 新增第九章 |
| **商业计划书"所有数据必须有来源"** | 市场数据/财务预测无依据 | 加"据XX报告"或改定性描述 |
