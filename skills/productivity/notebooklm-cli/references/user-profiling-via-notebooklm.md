# User Profiling via NotebookLM — 从知识库构建用户画像

> 2026-05-26 实战验证：通过 71 个 NotebookLM 项目，构建杨晓凯完整画像（身份/哲学/方向/偏好）
> 触发情景：用户说"理解用户非常重要" → 主动扫描知识库

## 触发条件

当用户暗示或要求你更好地理解他们时（如"理解用户非常重要"、"了解我"、"我的个人介绍在XX"），不要只依赖现有 MEMORY/USER 文件——**主动扫描用户的 NotebookLM 知识库**，那里通常有比对话更完整的记录。

## 工作流

### Phase 0: 摸底（5秒）

```bash
notebooklm list
```

快速过一遍笔记本标题，识别三类项目：

| 类别 | 特征 | 目标 |
|:-----|:------|:-----|
| **个人简介** | 标题含"个人介绍"/"个人档案"/"about me" | 直接提取身份信息 |
| **哲学/方法论** | 含"碳硅"/"超级个体"/"宣言"/"哲学" | 理解驱动力 |
| **核心研究** | 用户主方向的项目 | 验证研究方向 |
| **工作管理** | 含"工作流"/"规范"/"铁律"/"奖励办法" | 提取工作偏好 |

### Phase 1: 并行深度探查（核心步骤）

使用 `delegate_task` 并行探查多个 NotebookLM 项目。按探查目标分组，每组一个子 agent：

```python
# 三组并行模板
[{"goal": "用户身份/背景/经历 — 从NotebookLM个人介绍项目",
  "toolsets": ["terminal"]},
 {"goal": "用户核心技术方向和学术追求",
  "toolsets": ["terminal"]},
 {"goal": "用户工作方式偏好和个人驱动力",
  "toolsets": ["terminal"]}]
```

> ⚠️ **delegate_task 跨项目并行安全**：NotebookLM 不同项目（`use <id>`）有独立会话，跨项目并行不会串话。但**同一项目**的并行 `ask` 会串话（陷阱 #14）。

### Phase 2: 关键探针模板

```bash
# 个人介绍项目探针
notebooklm use <project_id>
notebooklm ask "请告诉我这个笔记本中关于我的个人介绍、背景、身份、经历等所有信息。逐个源文件检查"

# 哲学/驱动力项目探针
notebooklm ask "分析本项目想表达的核心哲学和方法论"

# 研究项目探针
notebooklm ask "这个项目的核心研究方向是什么？有哪些已发表论文？(只列作者包含我的)"

# 工作管理项目探针  
notebooklm ask "这个项目包含了哪些工作规范或偏好？"
```

### Phase 3: 信息萃取维度（四大支柱）

从 NotebookLM 输出中提取以下四类信息：

#### 1️⃣ 身份与背景
- 专业/职称/职务
- 教育经历
- 人才称号
- 社会兼职
- 团队信息

#### 2️⃣ 哲学与驱动力
- 核心信念（如碳硅共生）
- 终极目标
- 方法论（如超级个体）
- 临床驱动力来源

#### 3️⃣ 研究方向
- 核心技术方向（清单式）
- 非本人方向（ADHD 等合作者项目 → 标注清除）
- 产出状态（论文、专利、软著、设备）

#### 4️⃣ 工作偏好与约束
- 流程铁律（顺序、工具、质量门）
- 输出格式偏好（结构化 > 散文）
- 设计偏好（色值、字体、布局）
- 协作方式（自驱、不确认、不道歉）

### Phase 4: 写入 USER.md + MEMORY.md

将萃取的信息写入：

**USER.md** — 结构化用户画像（who they are）
```markdown
## 👤 用户画像 · 姓名（邮箱）

### 身份与角色
- ...

### 核心哲学
- ...

### 核心方向
1. ...
```

**MEMORY.md** — 跨会话注意事项（how to work with them）
- 工作流顺序铁律
- 设计/格式偏好
- 合作者归属标注
- 论文/产出数据声明纪律

### Phase 5: 请用户验证

输出更新后的 USER.md 全貌，请用户确认：
- "有没有偏的？漏的？不想我记住的？"
- 用户纠正 → 立即 patch，不道歉

## 已知陷阱

| 陷阱 | 表现 | 修复 |
|:-----|:------|:-----|
| **身份推断偏差** | Gemini 从合作者项目推断用户身份（如 ADHD → 推断为儿童保健科） | 必须在 USER.md 中清晰标注"合作者项目，非本人方向" |
| **个人简介不完整** | 一个 Notebook 不一定包含所有信息 | 并行探查多个项目，综合交叉验证 |
| **Q&A 时间长** | 每个 ask 需 60-90 秒 | 3 组并行 delegate_task 可控制在 2 分钟内完成 |
| **Gemini 幻觉** | 可能做出合理但不正确的推断 | 输出后必须让用户确认，重点纠正身份和方向 |
| **USER.md 容量限制** | 约 1,375 字符软限制 | 优先保留：身份/方向/铁律/偏好；哲学可用短标语 |

## 对比其它方法

| 方法 | 信息来源 | 速度 | 深度 | 适用场景 |
|:-----|:---------|:----:|:----:|:---------|
| **NotebookLM 扫描** | 知识库 | 中 | 🔥深 | 有丰富知识库的用户 |
| **对话中积累** | 自然交互 | 慢 | 🟡中 | 长期渐进了解 |
| **直接问用户** | 用户 | 快 | ✅准 | 紧急/简单场景 |
| **GitHub/论文档案** | 公开数据 | 中 | ✅准 | 仅限学术产出 |
