# 会话案例：竞赛项目工作区搭建 → 反思抽象技能凝练

> 这是 `reflexive-abstraction` 技能首次应用的完整记录。
> 从"为一个竞赛项目搭建工作区"的实操中，
> 提炼出"把项目经验凝练为技能"的通用方法论。

---

## 上下文

- **项目**: Synthos — 全球数智教育创新大赛 AI for Medicine 赛道
- **用户**: 杨晓凯，INSTITUTION_NAME_PLACEHOLDER
- **问题**: 每次会话都重新读取项目材料，效率低，无持续进步

## 会话演进：从操作到反思的3步跳变

### 第1步：用户提出需求（操作层）

用户说： "改变优化策略——应该是一个项目，然后持续优化，一直到满足高质量要求。"

**响应**: 我搭建了项目工作区：
```
.hermes/projects/competition-synthos/
├── PROJECT.md        ← 项目宪法（不变事实）
├── QUALITY.md        ← 质量矩阵（L1-L4 逐级目标）
├── DELIVERABLES.md   ← 交付物清单（状态跟踪）
└── CHANGELOG.md      ← 优化日志（变更记录）
```

### 第2步：用户提出哲学洞察（思想层）

用户说： "自我进化凝练技能应该是我们的核心任务。提高思想高度是我们优于其他项目的最强特征。如何从做项目的过程中提取规律、凝练成技术？"

**洞察**：这不是要完成某个任务，而是要建立**从经验中学习的元能力**。

### 第3步：反思抽象（meta层）

我识别出：
1. 刚才做事的模式（搭PROJECT.md/QUALITY.md/CHANGELOG）本身就是可复用的工作流
2. 用户的思想洞察（从经验中提炼）是一个缺失的技能类别
3. 这个技能应该与 `skill-absorption`（外部吸收）互补，形成完整的"向外看+向内看"体系

**产出**: `reflexive-abstraction` skill

## 关键抽象过程

| 具体（这次会话） | 抽象（通用形式） |
|:----------------|:-----------------|
| Synthos 竞赛材料 | 任何需要持续优化的项目 |
| PROJECT.md | 项目宪法（不变事实→减少认知加载） |
| QUALITY.md | 质量矩阵（迭代目标→避免无方向感） |
| CHANGELOG.md | 优化日志（变更追溯→后视镜） |
| 用户说"改变策略" | 用户纠正 → 最强烈的技能信号之一 |
| 用户说"凝练技能是核心任务" | 用户提出哲学洞见 → 必须形式化 |
| 我问"这是什么类的技能" | 边界检查 → 确认是新类不是已有类的变体 |

## 思想提升记录

### 问题1：为什么"项目宪法+质量矩阵+优化日志"的结构有效？

**操作层回答**：因为不再需要每次从头读取，质量缺口一目了然。

**原则层回答**：因为它实现了**认知卸载**（cognitive offloading）——把"记住项目状态"的任务从工作记忆转移到持久化文件。这是知识管理的第一原理。

**哲学层回答**：因为它让项目的进化轨迹可见。可见的进化才是真正的进化。看不见的进化等于没有发生。

### 问题2：为什么"从经验中提取技能"是核心任务？

**操作层回答**：这样下次再干类似项目就能直接上手。

**原则层回答**：因为经验的价值不在于经验本身，而在于从经验中提取的规律。规律是比其他任何资源都更高效的认知压缩形式。

**哲学层回答**：因为这是碳硅共生的本质——人类提供实践和判断（做什么、为什么），Agent 承担观察和抽象（看到什么模式、如何形式化）。人类不做 Agent 能做的抽象工作，Agent 不做人类需要判断的工作。
