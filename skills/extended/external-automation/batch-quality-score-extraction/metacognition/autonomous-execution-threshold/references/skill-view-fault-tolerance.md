# Skill View Fault Tolerance Protocol

> 2026-06-12 会话记录：对数百个不存在的技能名调用 `skill_view()` 陷入无限循环。
> 根因：`skills_list` 返回完整清单 → 对每个技能名调用 `skill_view()` → 不存在的返回错误 → 继续尝试下一个 → 数百个不存在 → 数百次失败调用。
> 教训：`skill_view` 是只读的，但每次调用消耗 token；没有失败计数上限。

## 安全规则

| 规则 | 值 |
|------|----|
| 连续失败上限 | 10 次 |
| 失败后行为 | 立即停止，报告失败率 |
| 最大技能查询 | 15 个/轮（超出则用 `skills_list` 判断） |
| 空名/短名 | 直接跳过，不计入失败 |

## 错误模式

**症状**：对 `skill_view("git-tag-verify-forty-three")` 这种明显不是有效技能名的东西进行调用。

**根因**：将 `skills_list` 的完整输出直接映射为 `skill_view` 输入，未做任何过滤或验证。

**修复**：
1. `skill_view` 前检查技能名是否在 `skills_list` 返回的已知列表中
2. 遇到失败时计数，连续 10 次失败立即停止
3. 对长列表使用批量策略：只查询前 N 个 + 最后 N 个，中间跳读

## 验证清单

- [ ] 技能名来自 `skills_list` 的 `available_skills` 列表？
- [ ] 连续失败次数 < 10？
- [ ] 本次查询的技能总数 ≤ 15？
