# Conversation → Memory

> 凡记必择，过期不录。宪不可违，记以长养。

## 与 memory-enhancement 的分工

| 维度 | conversation-to-memory（本技能） | memory-enhancement |
|:-----|:---
  io_contract: input: ['session_summary: str, memory_usage: float -> saved_entries: list[str], replaced_entries: list[str], remaining_space: int', 'output: ['saved_entries: list[str], replaced_entries: list[str], remaining_space: int']
----------------------------|:-------------------|
| 角色 | 筛选层 | 结构化层 |
| 输出 | `memory` 工具（2.2KB简记） | 知识图谱节点（Obsidian/文件系统，含S,D,R） |
| 频率 | 每会话 | 每复杂任务 + 每周cron巩固 |
| 保留期 | 7天有效，自动过期 | 永久，按FSRS衰减管理 |
| 知识形式 | 压缩文本 | (S,D,R)状态+图谱连接+增量阅读态 |
| 宪法检查 | ✅ 有 | 间接（通过本技能） |

**协作流**：
```
本技能筛选 + 三问 → 低价值→memory（L2工作层）
                    ↘ 高价值（有发酵潜力）→ memory-enhancement 建知识节点（L4长期层）
```

## 核心理念（文言）

| 白话 | 文言 | 义 |
|:-----|:-----|:----|
| 记忆不是日记，是系统生长营养 | **记非日记，长养为要** | 每条记忆为系统生长提供营养 |
| 7天后还有用才存 | **七日无益则不存** | 临时状态让session_search去找 |
| 记忆不能覆写宪法 | **宪不可违，记在其下** | 记忆在CONSTITUTION之下 |
| 陈述事实，非指令 | **述实不令** | "用户偏好XXX"而非"必须XXX" |
| 压缩合并同类项 | **并类压缩** | 多条同类→一条概括 |

## 宪法护栏

| 原则 | 记忆可增强 | 记忆不可削弱 |
|:-----|:----------:|:------------:|
| 碳硅共生 | ✅ 记录用户决策偏好 | ❌ 不能授权AI替代人决策 |
| 自进化最高 | ✅ 记录新吸收来源 | ❌ 不能退化进化优先级 |
| 原子边界精确 | ✅ 记录技能关系 | ❌ 不能创建重叠技能 |
| 认识论原则 | ✅ 记录可靠性来源 | ❌ 不能编造数据 |
| MIT开源 | ✅ 记录许可证兼容 | ❌ 不能闭源核心 |

### 冲突自检

保存前自问：
1. 这条记忆是否指导我违反CONSTITUTION不可修改条款？→ 是则**不存**
2. 是否与P0-P7原则冲突？→ 轻微则标记"本地例外"，严重则**不存**
3. 是否会改变我的核心行为？→ 是则触发**project-experience-distillation**，不是记忆任务

## 主动用户画像（Active User Profiling）

> 「理解用户非常重要」— 不能等用户告诉你是谁，要从能获取的信息源主动推导。

当用户表达「理解」/「懂我」/「个人化」相关诉求，或你发现现有画像过于单薄时，主动从外部知识库挖掘。

### 可选的外部知识源（优先级排序）

| 优先级 | 来源 | 获取手段 | 信息价值 |
|:------:|:-----|:---------|:---------|
| ★★★ | NotebookLM项目 | notebooklm list → 遍历 → delegate_task分路并行分析 | 研究方向、项目格局、论文产出 |
| ★★★ | 项目文件系统 | ls关键目录(GitHub repos、论文文件夹、代码) | 工程能力、成熟度、实际产出物 |
| ★★☆ | GitHub Profile | gh repo list 或 GitHub API | 开源项目、合作网络、贡献记录 |
| ★★☆ | 已发表论文 | notebooklm ask按作者过滤 | 学术产出、合作者、期刊偏好 |
| ★☆☆ | 会话历史 | session_search回顾过去交互 | 纠错历史、偏好模式、已解决问题 |

### NotebookLM并行画像流程

不要逐个问，用 delegate_task 并行扫：

```
Step 1: notebooklm list → 获取全部笔记本（Owner优先）
Step 2: 按领域分组（核心技术/临床/方法论/个人成长/系统文档）
Step 3: delegate_task 3-4路并行分析：
  路1: 核心技术笔记本 → 研究方向+技术栈+创新点
  路2: 临床/应用笔记本 → 临床方向+疾病焦点+诊断痛点
  路3: 方法论/系统笔记本 → 哲学观+工作方法论+工具偏好
  路4: 其他笔记本 → 兴趣广度+合作项目+非核心方向
Step 4: 交叉验证 — 不同笔记本中矛盾信息标记待确认
Step 5: 更新 USER.md（声明式事实）+ MEMORY.md（注意事项）
```

### 交叉验证原则

| 信号 | 处理 |
|:-----|:------|
| 多个笔记本指向同一方向 | ✅ 高度可信，直接采纳 |
| 单项目线索 | ⚠️ 标记合作者项目或待确认 |
| 子agent推测与现有USER.md冲突 | ⚠️ 向用户提问确认后再更新 |
| 专业身份从单特征推断 | ⚠️ 多源交叉，不依赖单点 |

### 常见陷阱

| 陷阱 | 表现 | 修复 |
|:-----|:-----|:-----|
| 单点过推 | 从ADHD项目推断为儿保科 | 交叉验证核心5方向维度 |
| 合作者项目误标为主方向 | NotebookLM中合作者项目占大量源 | 标记合作者项目非本人方向 |
| 子agent幻觉 | delegate_task从NotebookLM数据错误推断用户身份（如从ADHD推断为儿科医生，实际是神经内科） | 合入USER.md前必须向用户展示关键推断并等待确认；校对多源交叉验证；子agent职责限定在\"数据采集\"而非\"身份推断\" |
| 忽略用户纠正 | 修正后仍按旧画像运行 | 立即更新+附带原错误文本比对；记录到memory让下次会话不再犯 |
| 一步到位不确认 | 一轮推断完成后直接写入USER.md，未给用户审阅机会 | 先展示关键推断，等用户说\"对\"或纠正后再写入永久记忆

### 画像维护节奏

| 时机 | 动作 |
|:-----|:------|
| 用户说理解/懂我/纠正专业身份 | 立即执行完整主动画像 |
| 每认识用户30天 | 快速扫描，更新过期信息 |
| USER.md/MEMORY.md接近满 | 压缩合并，替换低价值条目 |
| 用户抱怨你又不了解我了 | 重新做一轮完整画像 |

## 触发条件

每次会话结束（尤其是复杂任务后）：

- [ ] 用户说了"记住"/纠正了你？
- [ ] 发现了用户偏好（语言/风格/习惯）？
- [ ] 发现了环境细节（路径/端口/配置）？
- [ ] 现有记忆过期了（项目状态/API变化）？
- [ ] 记忆接近满（>2,000 chars）？

## 四步流程

### 第1步：收集候选

| 类别 | 示例 | 优先级 |
|:-----|:-----|:------:|
| 用户偏好/纠正 | "不要用emoji"、"中英混合" | 🔴 最高 |
| 环境/项目细节 | 路径/端口/配置文件 | 🟡 中 |
| 工作流/方法 | "先测试再写代码" | 🟢 低 |

### 第2步：三问过滤

| 问题 | 抛掉 | 保留 |
|:-----|:-----|:-----|
| 7天后能产生新知识/技能吗？ | 今日故障→**不存** | 路由/端口/偏好产生新认知→**存** |
| 能帮助系统理解自己的生长轨迹吗？ | 已完成任务→**不存** | 用户风格/项目规范显示生长方向→**存** |
| session_search能找到？ | 还能查到的→**不存** | 查不到的→**存** |

### 第3步：凝练压缩

```
❌ 指令式: "Always use cyan for covers"
✅ 陈述式: "Cover: teal/cyan accent (0,188,212), dark navy bg"

❌ 指令式: "Never fabricate data"
✅ 陈述式: "Data must be real and verifiable from project sources"
```

**技巧**：合并同类项、去冗余、用连接符替完整句子

### 第4步：空间管理

>2,000 chars 时：
1. 替换过期条目（项目状态/API变化）
2. 按价值排序：用户偏好 > 环境细节 > 方法 > 过期项目
3. 删除已完成进程信息（PR merged等→session_search）

## 循环迭代自检

### 动灵记忆三问

每条记忆在保存前，标注以下三个维度的回答：

| 维度 | 问句 | 回答示例 |
|:-----|:-----|:---------|
| **生长方向** | 这条记忆支持系统哪个生长方向？ | 增强对用户偏好判断的准确性 |
| **框架维度** | 与7+1框架哪个维度相关？ | P2(原子边界精确)：记录新工具路径 |
| **发酵潜力** | 这条记忆能发酵（转化为新能力/新技能）吗？ | 可沉淀为skill:记住用户路径偏好 |

**原则**：不能回答这三问的记忆，在生长视角下价值存疑。但暂不为此设硬性删除，而是作为优先级排序依据。

```
写记忆 → 自检(宪法对齐+7天有效) → 合格? → 再读一次 → 合格? → commit
    ↑                                             ↓
    └─────────────── 不合格则修改 ──────────────────┘
```

## 验证

- [ ] 每条记忆在未来会话中有用
- [ ] 无"已完成任务"类信息（归session_search）
- [ ] 无指令式语言（都是陈述事实）
- [ ] 总字数 < 2,200
- [ ] 宪法对齐：无违反不可修改条款
- [ ] 非核心行为变更（那是skill更新）

## 陷阱

1. **记录已完成任务**：PR#42 merged、batch done→7天后是噪音
2. **指令式语气**："Always use XXX"→被误解为硬规则，改为"User prefers"
3. **分散同类项**：同项目路径分多条→合并
4. **跳过宪法检查**：记忆不能绕过CONSTITUTION

## 数据流

本原子的输入由上游编排者（task-router 或其他原子）提供，输出直接返回给调用者。

## 错误处理

- 输入不合法 → 返回错误状态码 + 原因说明
- 执行失败 → 返回 error_state 节点，不产生半成品输出
- 超时 → 返回 timeout 标记，不阻塞调用者
