# 质量报告数据时效性陷阱

> 2026-06-25 实战教训：HCS-3WT 质量报告生成时 PDF 覆盖率仅 31.3%，但后续批量下载补全至 100%。报告的过时数据误导了用户判断。

## 问题

质量报告的生成和 PDF 下载往往是**异步**的：
1. 报告先生成（基于当时的 PDF 收藏状态）
2. PDF 批量下载后完成（覆盖率从 31%→100%）
3. 报告中的数据变得过时
4. state.json 仍标记 PASS，没有自动提示"报告数据已过时"

## 预防措施

### 1. 报告头部必须标注数据日期

```markdown
| 数据项 | 评估日期 | 当前值 |
|:-------|:--------:|:------:|
| PDF 覆盖率 | 2026-06-25 04:50 | 31.3% → 后补至 100% |
| 数值可追溯率 | 2026-06-25 04:50 | 82.5% |
```

### 2. PDF 批量下载完成后必须更新报告

```bash
# 流程：
# 1. 初始报告生成（含低覆盖率标注）
# 2. 启动 PDF 批量下载（background+notify）
# 3. 下载完成后重新生成报告的 PDF 矩阵部分
# 4. 更新报告中的覆盖率数据
# 5. 更新 state.json 的 timestamp
```

### 3. 报告中的"不可追溯"项需区分"永久缺失" vs "待补充"

| 类型 | 说明 | 处理 |
|:-----|:-----|:-----|
| 永久缺失 | 实验已不可复现，原始数据丢失 | 标记为 P0/P1 待修复 |
| 待补充 | PDF 待下载、实验待重跑 | 在报告中标注"待补充"而非"缺失" |

### 4. 报告应支持增量更新

对已有质量报告的论文，重新生成报告时应：
1. 复制原报告为 `report-v1.md`
2. 更新数据后生成 `report-v2.md`
3. 在 state.json 中记录最新报告版本
