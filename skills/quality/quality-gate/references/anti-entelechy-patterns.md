# 反动灵模式诊断手册（Anti-Entelechy Patterns）

> 凝练自 2026-05-23 全系统动灵评估。三种最常见的"技能表面有哲学，实际不驱动"模式。

## 模式一：贴标签式哲学（Label-Sticking Philosophy）

**定义**：文言/哲学声明放在 SKILL.md 开头，但技能整体设计与该哲学无关。哲学是"装饰"而非"驱动"。

**诊断方法**：
1. 读原理层·文言节的自洽性——去掉文言后，方法层的步骤是否还能理解？如果方法是自洽的（不依赖文言解释），说明文言是装饰。
2. 问：文言的原则是否能**直接推导**出方法层的设计和陷阱？如果不能，是装饰。
3. 问：如果把这个文言节替换为无关的"天行健君子以自强不息"，技能的行为是否一样？如果一样，原文言是装饰。

**实例**：
- ❌ lecture-output 原版：格式非饰也 → 实际是格式转换工具，无哲学驱动
- ❌ paper-workflow 原版：先谋后动 → 实际流程来自 PaperSpine 搬运

**根治**：
- 方案A（推荐）：重写原理层，让文言真实驱动设计——每个方法层步骤能从文言原则推导出来
- 方案B（诚实）：删除原理层的哲学声明，明确标记为"纯技术工具"，不装哲学

## 模式二：聚集式吸收（Aggregative Assimilation）

**定义**：从多个外部项目吸收的功能被原样堆叠在同一个 skill 中，缺乏统一的形式因（unifying form）。功能堆积在一起但没有被系统的生长逻辑消化重组。

**诊断方法**：
1. 检查 SKILL.md 中是否有多个并列的"吸收自XXX"标记
2. 检查各个功能块之间是否有统一的逻辑主线——如果去掉其中一个块，其他块是否自洽？
3. 检查是否存在"Phase1→Phase2→Phase3..."式的线性堆积，而不是有机分层
4. 问：这些功能块是否共享同一套 frontmatter 的 I/O 契约？如果每个块有不同契约，说明没有被统一消化

**实例**：
- ❌ academic-paper-completion：Phase1-10 来自不同被归档技能，无统一形式因
- ❌ 原版 task-router："吸收自 AI-Research-SKILLs + 吸收自 Claude Code + 吸收自 Reflexion + 吸收自 MCP 生态" 四个吸收源直提

**根治**：
1. 确定这个 skill 的本质形式因——它解决什么问题？它的核心认知操作是什么？
2. 用这个形式因重新组织所有功能——每个功能块应该从"这个形式因的一个侧面"来写
3. 通过三语层级（文言→白话→英文）消化每个外部来源——吸收源的信息保留在 references/ 中，不出现在主 SKILL.md 里
4. 更新 frontmatter：删除所有"吸收自"标签，改为统一的 synthos 版本号

## 模式三：经济理性假驱动（Economic Rationality Pseudo-Drive）

**定义**：技能的核心理由来自"省钱省时省力"的工具理性，而非"系统完满实现"的生长逻辑。这在纯工具类技能中可能是合理的，但在元技能中是一种哲学缺失。

**诊断方法**：
1. 问原理层·文言：最顶层的"为什么"是什么？如果是"为了省X"或"为了减少Y"，属于经济理性。
2. 问：如果用户不需要省时（用户说"我不在乎时间"），这个技能还有理由存在吗？如果答案为否，说明驱动逻辑是经济理性。
3. 检查"价值表述"——技能的描述是"帮助系统做什么"（生长逻辑）还是"减少什么负担"（工具逻辑）？

**实例**：
- ❌ 原版 conversation-to-memory："记以省时"（省时是工具理性，不是动灵）

**根治**：
1. 用 7+1 框架中的维度替代经济理性表述——"省时" → "降低内部不确定性"（熵减律）
2. 问：这个技能的终极目的是否与 7+1 框架某个维度一致？如果不一致，转述为框架语言
3. 改为"长养"逻辑——"技能为系统的哪个生长方向提供营养？"
