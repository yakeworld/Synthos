# 工程学技能标准格式 — 反模式与正确做法

## 反模式: 按主题组织章节

创建SRE/容量/混沌等工程学技能时，按技术领域组织章节:
- "## 核心哲学" → 这是**错误**的
- "## 可观测性三支柱" → 这是**错误**的
- "## SRE基础框架" → 这是**错误**的

## 正确做法: Synthos标准格式

每个技能必须严格遵循:
1. **YAML头**: name / description / version / allowed-tools / metadata.synthos / triggers
2. **## 触发条件**: ≥3个具体触发场景
3. **## 执行步骤**: 编号 + 粗体，每步含具体命令/操作
4. **## Pitfalls**: 具体陷阱 + 修复方案
5. **## 参考**: 文档链接或文件引用

## 验证 checklist

- [ ] YAML头完整: name, description, version, allowed-tools, metadata.synthos, triggers
- [ ] ## 触发条件: ≥3个场景
- [ ] ## 执行步骤: 编号+粗体，含具体命令
- [ ] ## Pitfalls: 具体陷阱+修复
- [ ] ## 参考: 链接或文件引用
- [ ] 没有"## 核心哲学"等非标准章节

## 根因

按内容主题组织是人类自然思维。但Hermes Agent技能是**可执行操作手册**——不是知识库文章。第一步永远是"什么时候执行"，第二步永远是"具体怎么执行"。
