## IO_CONTRACT

- **input**: `topic: str, style: str, source_material: str` — 帖子主题、风格（技术分享/科普/商业）、素材来源（文件路径或主题描述）
- **output**: `result: dict` — 包含正文、标签、封面文案、排版建议、配图清单的完整帖子

## 核心原则

1. **小红书风格**：口语化、分段短、留白多、每段3-5行
2. **封面优先**：先想封面文案，再写正文
3. **标签齐全**：技术类10-15个标签，科普类8-12个
4. **内容真实**：不编造数据、不虚构案例、所有claim可验证
5. **Synthos归属**：介绍Synthos工具/技能时，必须明确标注"Synthos自动化科研系统"归属 + GitHub链接 (github.com/yakeworld/Synthos)，不可模糊为通用"AI工具"。同时说明该技能在skills/下的路径。
6. **视觉密度**：每篇小红书帖子至少生成**3-5张配图**，不少于3张（封面+2张正文配图）。仅1张封面是不够的。优先：封面(call-to-action) → 架构/流程图(展示原理) → 数据图(真实结果截图或矩阵)。
7. **正向表述优先**：科普/分享类帖子避免以撤稿、失败、负面案例为主线内容。如需引用对比，以"我们避免了什么问题"的正向表述替代"某某论文被撤了"的负面渲染。

### 用户偏好的内容格式（2026-06-26 校正）

- **标题 = 强论断句**：`质检是AI辅助科研最重要技能` ✓，不用问句 `你的论文经得起质检吗？` ✗
- **品牌必带**：明确说明这是 Synthos 系统的技能，附 GitHub 链接 `https://github.com/yakeworld/Synthos`
- **技能名称**：写明对应的技能名称（如 `quality-gate`）
- **每帖至少 3-5 张配图**：封面(1200×1600) + 架构图 + 数据图 + 生态图，深色科技风

### 配图生成

**每篇帖子至少3-5张配图。** 只生成1张封面是不够的。

推荐配图组合：
| 图序 | 内容 | 规格 |
|:----|:-----|:----|
| 封面 | Title card，call-to-action文案 | **1080×1440 (3:4)** 或 1200×1600 |
| 图2 | 架构/流程图（展示原理，如8章结构） | 1080×1440 或 1200×1200 |
| 图3 | 数据矩阵/对比图（展示效果对比） | 1080×1440 |
| 图4 | 生态/归属图（Synthos技能全景） | 1080×1440 |

**全部配图必须在一个批处理脚本中生成** — 不要分多次调用 Pillow。写一个`/tmp/xhs_all_figures.py`，包含多个 `make_*()` 函数统一调用：
```python
def make_cover(): ...
def make_structure(): ...
def make_comparison(): ...
# 统一执行
make_cover()
make_structure()
make_comparison()
```
**禁止分多次写独立脚本执行** — 同一篇帖子的所有配图必须一个脚本批量输出。修改配图时修改该脚本重新执行即可。

配图规范：
- 深色科技风：#0F172A 底 + #3B82F6 蓝 + #F59E0B 金 + #1E293B 卡片
- 分辨率：**1080×1440 (3:4)** 为小红书标准封面比例，优先使用
- PNG 格式
- 底部统一加 footer: "Synthos quality-gate | github.com/yakeworld/Synthos"
- 封面必须醒目显示 Synthos 标识和技能名称

## 生成前验证

**必须先读技能/代码的实际能力，再写内容。** 不要凭记忆或推测描述工具能力。

流程：
1. 先 `skill_view()` 加载相关技能 SKILL.md
2. 检查 `scripts/` 目录下的实际脚本（grep 统计功能）
3. 必要时读核心脚本提取真实数据源/函数名
4. 基于真实代码生成帖子内容

错误示范：凭记忆说"有7个下载源"（实际20+）
正确做法：先读 `pdf-download-racing/SKILL.md` + `scripts/` 下的全部脚本，统计实际功能

## Pitfalls

1. **标题风格偏好** — 技术分享类帖子标题用**陈述句/断言式**而非问句。用户明确纠正：`"质检是AI辅助科研最重要技能"` ✅，`"你的论文，经得起质检吗？"` ❌。用"是什么/怎么做"替代"你知道吗/你做过吗"。封面标题同理。
2. **不要直接用 xhs 发帖技能发 X/Twitter** — 这是两个平台的独立内容
2. **小红书不支持代码高亮** — 技术帖子需要截图代码或简化代码展示
3. **小红书对长文有限制** — 超过2000字需分段发布
4. **配图优先** — 有图的帖子流量远高于纯文字
5. **Synthos 归属必须出现在正文开头** — 在介绍任何 Synthos 组件/技能时，第一段或第二段就标注"由 **Synthos** 自进化科研平台驱动"，而不是只在结尾或页脚出现。用户期待打开就看见归属。同时提供 GitHub 仓库链接（github.com/yakeworld/Synthos）和技能名称。开场出现 + 结尾重申 = 最佳实践。
6. **每帖必须加互动收尾** — 小红书平台特性决定了纯告知型帖子互动率低。正文末尾必须加一句互动钩子，如"你觉得这个思路怎么样？评论区聊聊👇"或"关注我，下期讲具体实现"。纯引流的结尾（仅放链接）效果远差于带互动的结尾。
7. **配图必须在单脚本中批量生成** — 同一篇帖子的所有配图（封面+结构+流程+对比）必须写在同一个 `/tmp/xhs_all_figures.py` 脚本中，用统一的 `make_*()` 函数一次性输出。禁止分多次写独立 Pillow 脚本执行。修改配图时修改该脚本重新执行即可。
6. **生成前必须验证来源能力** — 先读技能/代码，再写内容，不要凭记忆描述
7. **Pillow 圆角矩形** — 使用 `draw.rounded_rectangle(rect, radius=N, fill='#...')` 而非 `draw.rectangle` 配合 `radius` 参数。Note: `draw.rectangle` 不支持 `radius`；必须用 `draw.rounded_rectangle` 方法。
8. **配图路径约定** — 帖子素材统一存放在 `/home/yakeworld/桌面/social-media/xhs/<帖子名>/images/`，封面+架构图+流程图各一张。封面 1200×1600，深色科技风 (#0F172A + #3B82F6 + #F59E0B)。
9. **Pillow 脚本不存技能目录** — 生成脚本写到 `/tmp/xhs_*.py` 后执行，不存到 `references/`（生成代码非持久资产，每次配图内容不同）。
10. **配图使用 font 回退** — `/usr/share/fonts/truetype/dejavu/DejaVuSans-Bold.ttf` 存在则用，否则用 `ImageFont.load_default()`（无中文，但小红书帖子正文用 emoji+ASCII 代替）。
11. **负面内容禁区** — 科普/技术分享类帖子不得以撤稿、学术不端、失败案例作为主线内容。用户要求去掉时立即移除。可用"我们避免了什么"的正向表述替代"某某被撤稿了"的负面渲染。
12. **Synthos归属缺失** — 介绍Synthos任何组件/技能时，必须明确标注"Synthos自动化科研系统"归属 + GitHub仓库链接(github.com/yakeworld/Synthos) + 技能路径。不标注=不合格帖子。
13. **配图数量不足** — 每篇帖子至少生成3-5张配图（封面+架构+数据+对比+生态），不能仅1张封面。用单个`/tmp/xhs_all_figures.py`脚本批量生成所有图。
14. **帖子类型识别** — 生成前先判断帖子类型：技能介绍 vs 开源项目推广 vs 科普分享。不同模板的标题风格、品牌归属、链接方式不同。不当混用会导致帖子定位模糊。详见上方"开源项目推广模板"。

## 验证清单 · VERIFICATION

1. **输入验证**: 输入参数/文件/路径是否完整且有效
2. **过程验证**: 中间步骤/转换/计算是否正确
3. **输出验证**: 输出格式/内容是否符合预期
4. **边界验证**: 空输入、极大值、异常场景是否处理
5. **错误处理**: 失败时是否有明确的错误信息和恢复指引

> 每项验证必须可执行、可记录、可复现。验证失败时记录原因和修复。
