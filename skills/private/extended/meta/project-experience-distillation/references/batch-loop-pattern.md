# Batch Loop Pattern

> 2026-06-28 Cycle 189-190 批量自动循环实战提炼
> 适用场景：任何需要连续多轮自动化执行的循环场景（进化循环、批量数据修正、批量验证注入等）

## 模式定义

连续执行 N 轮自动化操作，每轮之间不等待用户输入，每轮结束后立即 commit。

## 正确执行顺序

```
for i in range(N):
    1. 运行诊断/探针 → 获取当前状态
    2. 检查继续条件 → 不满足则 break
    3. 执行本轮操作（修改文件/数据等）
    4. git add + commit 本轮所有变更
    5. 确认 git status --porcelain 返回空（clean state）
    6. 运行最终诊断（clean 状态下）
    7. 更新 state（精确计算，不可估算）
    8. 写入日志
    9. commit state + log
    10. 回到步骤 1（如果条件仍然满足）
```

## 关键约束

### 每轮必须 clean

每轮操作后的 git status 必须干净，否则下一轮的诊断会受 dirty 污染。

### 不可回滚

批量循环中每轮的 commit 都是累积的，不需要也不应该回滚。

### 条件检查在每轮开始时

每轮开始前必须重新检查继续条件（score、status、consecutive_healthy），不能只在第一轮检查。

### 终止条件

| 条件 | 行为 |
|:-----|:-----|
| no more items to process | 终止（如：所有技能已验证） |
| score < 0.85 | 终止 |
| consecutive_healthy >= 20 | 终止（burnout 保护） |
| 条件不满足 | 终止 |

## 实战案例

### Case 1: 批量验证清单注入（Cycle 189-190）

- 目标: 191 个 SKILL.md 全部添加验证清单
- 每轮操作: 35 个技能
- 实际执行: 2 轮（35+25=60 个验证，加上之前的 131 个 = 191/191）
- 第 3 轮（Cycle 191）因"no remaining unverified"提前终止
- 结果: 5-cycle 批量请求实际只用了 2 个周期

### Case 2: 批量示例添加（Cycle 191）

- 目标: 为缺少示例的技能添加示例章节
- 每轮操作: 35 个技能
- 实际: 8 个技能缺少示例，1 轮完成
- 示例覆盖率: 68 → 190/191 (35.6% → 99.5%)

## 陷阱

| # | 陷阱 | 正确做法 |
|:-:|:-----|:---------|
| 1 | 只在第一轮检查条件 → 后续轮次可能不满足 | 每轮开始前重新检查 |
| 2 | 批量循环中不 commit → dirty 累积 | 每轮操作后立即 commit |
| 3 | 批量循环超过 N 轮不终止 → burnout | 在循环内检查 consecutive_healthy >= 20 |
| 4 | 预期 N 轮但实际更少/更多时不更新计数 | 循环结束后确认实际执行轮数 |
| 5 | 不记录批量循环的总结 | 循环结束后用 project-experience-distillation 提取通用模式 |
