# 哲学→工程验证模式（Philosophy → Engineering Verification Pattern）

> 2026-05-23 | 7+1 东西合参框架原子验证实践
> 核心原则：**哲学框架必须能回溯到每个原子的工程约束，否则框架是空谈。**

## 适用场景

当哲学框架发生结构性变更后（新增维度、合并维度、改变层次），需要验证每个原子是否被框架正确约束。

## 验证流程

### Step 1: 提取约束矩阵

从框架文档中提取每维度的**工程约束**（不是哲学阐释，是"原子必须做什么"）：

```yaml
dimension_constraints:
  熵减律·生生之谓易:
    - "每个原子启动前问'这个任务在降低哪部分认识论熵？'"
    - "VER 输出不确定性降低量（验证前后置信度差值）"
  格物通理:
    - "输出结果时附'为什么成立'的理路"
  取象通变:
    - "类比必须输出映射对 + 适用边界 + 失效条件"
  天人合一:
    - "输出必须附观察者位置声明"
  经权度信:
    - "置信度可量化验证 + 声明所依之'经'（不变原则）"
  墨证求真:
    - "每个假设有可执行证伪测试 + 三表法验证"
  庄周观模:
    - "输出模型假设清单（≥3条假设+违反后果）"
  大道至简:
    - "每轮产出精简报告'能损掉什么'"
```

### Step 2: 追溯每个原子

对每个原子，逐条检查约束矩阵：

| 约束 | 原子 | 有? | 证据 |
|:-----|:----|:---:|:-----|
| 熵减: 启动前问"降低哪部分熵" | HYP | ✅ | Step 1.5 entropy_reduced |
| 熵减: 输出不确定性降低量 | VER | ✅ | uncertainty_reduction |
| 格物: 输出理路 | HYP | ✅ | reasoning_path |
| 取象: 类比映射对+失效边界 | HYP | ✅ | Step 3.5 disanalogy |
| ... | ... | ... | ... |

### Step 3: 分类缺口

| 层级 | 定义 | 处理 |
|:-----|:-----|:-----|
| 🔴 完全缺失 | 某维度在某原子中无任何对应 | 新增 Step / 修改 schema |
| 🟡 部分缺失 | 有对应但缺少关键字段或检查点 | 扩展现有 Step |
| 🟢 已有 | 完整实现 | 不动 |

### Step 4: 优先修复

修复顺序：🔴 熵减律（新维度）→ 🟡 各原子的"位置声明"（天人合一/庄周观模）→ 🟡 大道至简缺少检查点

### Step 5: 验证闭环

修复后重新加载验证：`skill_view(name)` 确认语法正确 + 约束矩阵再检查。

## 验证清单

- [ ] 约束矩阵已从框架文档提取
- [ ] 每个原子已逐条追溯
- [ ] 缺口已按 🔴/🟡/🟢 分级
- [ ] 已按优先级修复
- [ ] 修复后 skill_view() 可加载
- [ ] 约束矩阵覆盖率 ≥ 80%（新框架≥70%）

## 陷阱

1. **把哲学阐释当成工程约束** — "系统思维帮助看到全貌"不是工程约束。"VER 输出了 observer_standpoint"才是。区分：工程约束 = 原子输出的 schema 字段 + step 的检查点
2. **只检查核心原子** — HYP/VER/ARG 是重点，但 ACQ/EXT/ASC 可能也有缺口
3. **一次性全修** — 违反"一次一件事"原则。按优先级分多轮修复
4. **修完不验证** — 修改后必须重新加载确认语法正确
