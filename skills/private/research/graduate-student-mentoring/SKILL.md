# Graduate Student Mentoring — 研究生AI辅助科研培养

## 触发器

- 学生入组，需要制定培养方案
- 学生在研究过程中遇到瓶颈
- 需要调整研究方向/进度
- 开题/中期/答辩前需要检查进展

## 核心原则

1. **以战代练** — 不花专门时间学工具，用写论文驱动学习
2. **医学背景是优势** — 临床知识 + AI编程 = 独特竞争力
3. **第一篇求快不求完美** — 先建立完整流程信心
4. **深度优先** — 宁深勿广，聚焦三维眼动五大支柱

## 研究方向对齐

### 核心方向（全流程：假设→论文→投稿）
1. 瞳孔/虹膜分割
2. 眼球三维模型建模
3. 半规管空间姿态
4. BPPV虚拟仿真
5. VOR数字孪生
6. 算法组件（边缘检测、特征点提取、校准）
7. 公开眼动数据集分析/方法论审计
8. Synthos科研辅助系统开发
9. AI辅助教学

### 外围方向（仅提取研究空白和科学假设，不生成论文）
- 角膜/晶状体/玻璃体生物力学
- 泪膜/睑板腺
- 耳鸣/脑震荡等非眼动非数据集方向

## 三阶段培养架构

```
Phase 1: 工具筑基（第1周）
  ├── 目标：能用 AI 工具独立完成从想法到论文
  ├── 工具链：Hermes (agent) → OpenCode (编程) → Synthos (管线)
  └── 验收：复现一篇已有论文的核心图 + 一篇 mini-review

Phase 2: 数据获取（第2-4周，与写作并行）
  ├── 公开数据集优先（GazeCapture/Lund2013/OpenEDS）
  ├── 科室临床数据（vHIT/VNG视频眼震，需脱敏）
  └── 验收：完整分析管线跑通 + 第一张图

Phase 3: 论文写作（第5-8周）
  ├── 第一篇：Kappa角分布分析（公开数据集，6-8周）
  ├── 第二篇：三维 vs 二维瞳孔定位（第3-4个月）
  └── 验收：投稿
```

## 第一篇论文选题策略

| 选题 | 数据源 | 难度 | 周期 | 推荐场景 |
|:-----|:-------|:----:|:----:|:---------|
| 3D Kappa角分布特征 | GazeCapture/Lund2013 | ⭐⭐ | 6-8周 | 医学背景新手首选 |
| 3D vs 2D瞳孔精度对比 | 仿真+公开数据 | ⭐⭐⭐ | 8-12周 | 有基础编程能力 |
| 虹膜三维运动分析 | 科室vHIT/VNG数据 | ⭐⭐⭐⭐ | 10-16周 | 有临床数据渠道 |

## 投稿目标

- 第一篇：*Translational Vision Science & Technology* 或 *Journal of Eye Movement Research*
- 第二篇：*Biomedical Optics Express* 或中文核心

## 每周例会节奏

| 阶段 | 频率 | 重点 |
|:-----|:-----|:-----|
| 工具期（第1周） | 每天30min | 环境检查、方向确定 |
| 数据期（第2-4周） | 每周1次 | 结果讨论、方向把控 |
| 论文期（第5周起） | 每周1次 | 论文修改 |
| 长期 | 每月1次 | 进展汇报 |

## 风险控制

| 风险 | 应对 |
|:-----|:-----|
| OpenCode代码跑不通 | 导师用Hermes调Synthos技能库现有代码 |
| 公开数据集不够 | 切换至科室临床数据 |
| 统计分析不会 | 提供分析模板 + ChatGPT协助 |
| 论文写作慢 | 先写中文稿，导师改完再翻英文 |
| 方向发散 | 锚定五大支柱，定期回顾范围约束 |

## 学生自检清单（每周）

- [ ] 本周新增了多少行可运行的代码？
- [ ] 本周产出了几张图/几个Table？
- [ ] 是否理解了代码每一行在做什么？
- [ ] 有没有卡住超过2小时的问题？（有→马上问导师）
- [ ] 是否在AI生成的代码上做了自己的修改？

## 契约层 · BOUNDARY

**边界**：技能功能边界。

## 契约层 · IO_CONTRACT

**输入**：请求描述、上下文信息。
**输出**：执行结果、状态反馈。

## 验证清单 · VERIFICATION

1. **输入验证**: 输入参数/文件/路径是否完整且有效
2. **过程验证**: 中间步骤/转换/计算是否正确
3. **输出验证**: 输出格式/内容是否符合预期
4. **边界验证**: 空输入、极大值、异常场景是否处理
5. **错误处理**: 失败时是否有明确的错误信息和恢复指引


## 约束规则 · RULES

1. **输入约束**: 参数类型、范围、格式必须校验
2. **输出约束**: 返回值结构、编码、命名必须一致
3. **异常约束**: 错误信息必须包含上下文和恢复建议
4. **安全约束**: 不执行未验证的任意代码，不暴露内部状态


## Golden 集合 · GOLDEN SET

- **Golden Input**: 标准输入样本（覆盖正常路径）
- **Golden Output**: 预期输出（精确匹配或格式校验）
- **Golden Error**: 预期错误信息（覆盖失败路径）

> Golden 集合是测试的单一真理来源。所有改进必须通过 golden 测试。

> 违反规则的操作视为不安全，必须拒绝或隔离。

> 每项验证必须可执行、可记录、可复现。验证失败时记录原因和修复。
