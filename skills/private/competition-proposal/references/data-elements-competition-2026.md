# 2026 数据要素×大赛 浙江分赛 申报案例

## 场景概述

**项目**：基于自主智能体的医疗健康数据要素全生命周期治理与赋能平台（Synthos）
**申报单位**：温州市人民医院（事业单位）
**申报赛道**：医疗健康（非数据基础设施赛道）
**申报书类型**：DOCX模板 + PPT 路演

## 赛道评估过程

### 初始问题

用户发送了一份"数据基础设施赛道"的申报书模板。该模板包含大量企业导向的必填字段。

### 模板分析发现的问题字段

| 字段 | 要求 | 事业单位能否填 |
|:-----|:-----|:-------------|
| 营收/利润 | 2025年数据+增速 | 不能 |
| Token消耗量（亿/年） | 带增速 | 不能 |
| 融资阶段 | 天使/A/B/C/D轮 | 不能 |
| 企业规模 | 大/中/小微企业 | 不能 |
| 数据资产入表金额 | 万元+增速 | 不能 |
| 数据产品交易量 | 直接/交易所/数据商 | 不能 |

### 决策逻辑

1. 模板中20个赛道选项包括"医疗健康"——与用户的临床背景和专业方向完全匹配
2. 医疗健康赛道的评审关注：临床痛点、数据赋能诊疗、社会效益——实验室的积累正好对上
3. 事业单位在医疗健康赛道可正常参赛，财务字段可用"0"或"不适用"

### 结论

不报数据基础设施赛道，改报医疗健康赛道。同样的项目内容在不同赛道中竞争力完全不同。

## 项目定位策略

### 核心定位句

"一套从数据源头治理到AI模型产出、从文献知识提取到临床决策赋能的端到端数据要素智能治理系统"

### 三层叙事结构

| 层 | 内容 | 对应技术 |
|:--|:-----|:--------|
| 第1层：痛点 | 临床数据质量差、AI论文可复现危机、文献过载 | 数据泄露检测、G1-G7门控 |
| 第2层：方案 | AI Agent自动做数据治理+实验+验证+知识提取 | Synthos、CRISP-DM Helix、AKNE |
| 第3层：成效 | 跨数据集验证、多疾病领域复用、效率提升200倍 | PIMA/CDC/ED 30模型基准 |

### 翻译映射

每一次"技术术语到评审语言"的转换：

| 技术概念 | 申报书表述 |
|:---------|:----------|
| G1-G7质量门控 | 首创七级数据质量门控体系，确保每个实验步骤可追溯、可复现，从根源上解决可复现危机 |
| 30模型基准测试 | 已覆盖30个主流机器学习模型的系统性跨数据集验证，覆盖PIMA/CDC/ED三个独立数据库 |
| AI Agent自主进化 | 自省式智能体：能根据前次实验结果自动调整策略，识别数据集固有天花板，自动收敛到最优方案 |
| 非万能最优模型 | 不存在一个模型在所有医疗数据上最优——模型选择应与数据集特征（样本量、类别分布）相匹配 |

## 申报书结构

### 基本信息表格填写要点

| 字段 | 填写内容 | 说明 |
|:-----|:---------|:-----|
| 项目名称 | 基于自主智能体的医疗健康数据要素全生命周期治理与赋能平台 | 长但全面 |
| 赛道 | 已选：医疗健康 | 文本标记 |
| 项目简述 | 1000字内：痛点到方案到技术到成效 | 带量化指标 |
| 覆盖场景 | 医疗健康、公共服务、产教协同 | 多选 |
| 团队名称 | 智能医学与数据治理创新团队 | |
| 单位性质 | 事业单位 | 非企业 |
| 营收/利润 | 0 | 事业单位标明 |
| Token消耗 | 不适用 | 事业单位标注 |
| 数据资产入表 | 0 | 不适用 |

### 脚本封装

填充脚本采用4段架构：
1. 表格字段填充（set_cell函数遍历合并单元格）
2. 正文JSON内容加载与注入
3. 模板说明文字批量清理
4. DOCX保存

正文内容用JSON文件独立管理，避免Python字符串编码问题。

## 关键教训

1. 赛道选择大于内容质量 — 同样项目在错误赛道竞争力减半
2. 事业单位申报 — 财务数据填"0"或"不适用"是可以的，评审关注的是技术方案
3. 翻译优于解释 — 不要等技术评审"理解你的技术优秀"，要直接用他们熟悉的语言讲
4. 分离内容与代码 — JSON管理正文，Python管理填充，避免编码问题
5. 模板行数核实 — 政府模板的"35行"表格实际可能只有34行，先逐行检查
