# 政府座谈会准备模式（Symposium Preparation Patterns）

> 采集自 2026-06-09 脑机接口座谈会准备过程

## 一、材料采集与归档规范

### 1.1 多源材料识别

政府座谈会涉及的材料通常跨越多个本地目录：
- **本地项目目录**：`projects/bci-demo-base/` — 技术报告、政策摘要、发言提纲
- **人才申报目录**：`瓯越英才申报材料/` — 申报表、证明材料、PPT
- **方案原文**：用户通过聊天发送的 `.doc` / `.pdf` 文件（需先 `antiword` 或 `pdftotext` 提取）
- **缓存目录**：`.hermes/cache/documents/` — 聊天上传的文件

**关键陷阱**：本地文件名含特殊字符（如中文引号 `"`、括号 `()`）时，shell 命令需用单引号或双引号包裹完整路径，否则 `cp`/`pdftotext` 会报错。

### 1.2 归档原则

座谈会材料**不应分散在多个目录**，应统一归档到一个语义目录中：

```
bci-demo-base/
├── 3d-eye-tracking-bci-report.md     ← 核心报告（技术论证）
├── landscape/bci-base-landscape.md   ← 全国格局分析
├── policies/bci-policy-summary.md    ← 政策文件摘要
├── meeting-prep/
│   ├── speech-outline.md             ← 发言提纲
│   ├── speech-full.md                ← 完整发言稿
│   └── scheme-suggestions.md         ← 对方案的逐章修改建议
├── eye-bci/eye-bci-connection.md     ← 技术连接点
└── meeting-archive.md                ← 归档索引
```

**归档后必须**：更新 `digest.md` 或 `README.md`，建立快速导航。

### 1.3 旧会话材料查找

用户在旧会话中发送的材料，新会话**不继承上下文**。必须：
1. 先用 `find` 搜索本地磁盘（`-iname` 模糊匹配）
2. 检查 `.hermes/cache/documents/` 中的缓存
3. 检查 `projects/` 目录下的项目文件
4. 检查 `outputs/` 目录

## 二、方案审查模式

### 2.1 逐章审查结构

对政府方案的审查**必须逐章进行**，每章包含：

| 审查项 | 说明 |
|:-------|:-----|
| 现状描述 | 方案原文对该章的描述 |
| 建议补充 | 具体要增加的内容 |
| 建议修改文案 | 可直接替换的完整段落 |
| 理由 | 为什么这个修改必要 |

### 2.2 黑体标注原则

所有**建议直接写入方案正文的内容**必须用黑体（粗体）标识。这样用户可以直接复制粘贴，不需要再区分"建议"和"原文"。

### 2.3 表格必须结构化

建议写入方案的表格（如医院-病种-技术对应矩阵、重点攻关清单）必须使用 `| :--- | :--- |` 格式，且每列都有明确的表头。

### 2.4 新增章节建议

当方案的"主要任务"章节缺少具体项目/资金时，建议新增"重点攻关清单"章节：

```
五、重点攻关清单（2026-2028）
| 序号 | 攻关方向 | 牵头单位 | 合作单位 | 预期成果 | 资金需求 |
```

## 三、发言稿编写规范

### 3.1 时长控制

- 完整发言稿：**5-7分钟**（约 1800-2500 字）
- 开场白：**30秒**
- 核心观点：**30秒**
- 已有基础：**2分钟**
- 技术桥梁/方案建议：**2分钟**
- 结尾收束：**30秒**

### 3.2 内容层级

| 层级 | 内容 | 敏感度 |
|:-----|:-----|:-------|
| 有管线 | 手稿数量、算法代码数量、自研硬件 | ✅ 公开 |
| 有硬件 | 原型机规格、技术原理 | ✅ 公开 |
| 有数据资源 | CT数据量、数据集 | ⚠️ 仅提类型，不暴露具体数字 |
| 临床数据 | 门诊量、检查量、成功率 | ❌ 不写具体数字 |
| 论文发表 | 已发表 SCI | ✅ 可写 |
| 手稿/未发表 | 64篇已完稿 | ⚠️ 统一表述为"已完成手稿"，不用"已发表论文" |

### 3.3 关键话术

**被问"没发表论文"时**：
> "我们选择先做深度再做广度。有三样实打实的资产：64篇已完成手稿、10+项可部署算法、自研硬件原型。背后的科研操作系统本身就是系统化能力的证明。"

**被问"和天坛/无锡比优势"时**：
> "优势恰恰在于不和他们比。天坛做侵入式，无锡做TMS+DBS。我们做眼动-前庭——完全不同的赛道。"

### 3.4 数据暴露最小化原则

- 不写"年5000+门诊，3000+检查"
- 改为"已建立从临床到机制到原型的完整闭环管线"
- 不列具体数据集名称（OpenEDS、自建虹膜库等）
- 改为"已积累多种数据类型，可作为算法训练集"

## 四、Word 文档生成（python-docx）

### 4.1 环境陷阱

`sandbox` 环境的 `/tmp/` 路径中 `lxml` 可能被破坏。如果 `from docx import Document` 报错：
1. 先用 terminal 运行 `python3 -c "from docx import Document; print('OK')"` 验证
2. 如果失败，删除 `/home/yakeworld/.local/lib/python3.12/site-packages/lxml` 后重新 `pip install lxml`
3. 使用 terminal 直接执行脚本，而非 `execute_code` sandbox

### 4.2 文档结构

Word 文档必须包含：
1. 标题（居中）
2. 元信息（日期、会议、提出人）
3. 总体评价（3个缺口）
4. 逐章建议（每章：现状→建议补充→建议修改文案→理由）
5. 新增章节建议
6. 关键数据支撑
7. 发言策略

所有建议内容必须用 **bold** 格式。
