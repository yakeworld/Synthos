# 参政议政提案自动化管线

> 数据驱动，三段成文。一次科研产出，文章+提案双产出。

## 核心哲学

**证据比立场更重要。** 民进提案的生命力在于每项建议都有可追溯的数据支撑。不是"应该怎么做"而是"数据显示什么→因此建议什么"。

**论文即提案原料。** 每篇SCI论文中的政策含义、临床发现、数据洞察，都是高质量提案的现成素材。不从头写起，从已有研究中提取。

**思想在行动中建立。** 不必先写宣言再提案——你的实践已经在表达思想。多份提案自然聚合成支部品牌标签，比单独宣言更有说服力。策略路径：原型提案→被采纳/引起讨论→建立品牌→形成思想体系→方法论输出。

## 输出格式规范（⚠️ 用户偏好 — 2026-05-26 人才Token提案实战修正）

> **重要**：用户纠正了提案输出格式。以下规范必须严格遵守。

### 结构规范

| 要素 | 正确做法 |
|:-----|:---------|
| **标题** | `关于[主题]的提案` — 纯文字，不加粗，不换行 |
| **提案人** | `提案人：XXX  民进XX支部` — 纯文字，紧接标题下方 |
| **正文结构** | `1.基本情况` → `2.存在问题及其原因分析` → `3.对策建议` — 三段式 |
| **次级编号** | **不允许使用**（一）、1.、①等次级编号。全部以段落自然展开 |
| **段落组织** | 每个大段2-4个自然段落，段落之间空一行，不缩进 |

### 排版约束

| 元素 | 实际提案输出中 | SKILL.md/引导文档中 |
|:-----|:---------------|:--------------------|
| **表格** | ❌ 禁止使用 | ✅ 可用（指导性质） |
| **加粗** | ❌ 禁止使用 | ✅ 可用 |
| **项目符号** | ❌ 禁止使用 | ✅ 可用 |
| **流程图** | ❌ 禁止使用 | ✅ 可用 |
| **纯文字段落** | ✅ 必须使用 | ✅ 可用 |

**铁律**：交付给用户的提案文件必须是**纯文字段落**——无表格、无加粗、无项目符号、无流程图。所有数据引用嵌入正文，不列参考列表。

### 数据引用方式

❌ 错误（脚注/尾注/文献列表式）：
```
全国尚无面向个人的AI算力补贴政策[1]
...
[1] 北京、上海、深圳、杭州算力券政策对比，2025
```

✅ 正确（嵌入正文式）：
```
经查阅，全国尚无面向个人的AI算力专项补贴政策。北京、上海、深圳、杭州、浙江等地的算力券均以企业为补贴对象，个人通道完全缺失。
```

### 语言风格

| 要求 | 说明 |
|:-----|:-----|
| **直入主题** | 第一句即进入事实，不寒暄铺垫 |
| **数据驱动** | 每个论点有可追溯的数据支撑 |
| **一事一议** | 一份提案只聚焦一个问题 |
| **客观克制** | 不夸张、不煽情、不评价、不指责 |
| **精准量化** | 用数字代替模糊表述（如"年发5万+"而非"很多"） |
| **建议可操作** | 每条建议明确"谁来做、怎么做、何时做" |

```
标题：关于[主题]的提案

一、基本情况（背景+政策依据）
  ├── 问题概述：用数据说明问题的严重性和紧迫性
  ├── 政策背景：国家/省市相关政策文件（列出文件编号）
  └── 现实困境：现有政策/措施的不足

二、存在问题及其原因分析
  ├── 问题一：[具体问题] → 数据支撑 [N]
  ├── 问题二：[具体问题] → 数据支撑 [N]
  ├── 问题三：[具体问题] → 数据支撑 [N]
  └── 根因分析：为什么会出现这些问题

三、对策建议
  ├── 建议一：[可操作措施]（谁来做、怎么做、何时做）
  ├── 建议二：[可操作措施]
  └── 建议三：[可操作措施]
     → 每一条建议必须可落地、有责任主体、有可行性
```

## 结构化问题分析方法

### 三重失灵框架（多维度递进分析）

当分析一个系统性问题时，建议使用"三重递进失灵"结构：

```
第一重失灵：[最表层的问题]
  → 数据支撑：X%的状况不符合预期

第二重失灵：[比第一重更深层的问题]
  → 数据支撑：深层矛盾的数据验证

第三重失灵：[根因/结构性失灵]
  → 数据支撑：制度层面的真空或错配
```

**实战案例**（人才Token提案）：
| 层级 | 问题 | 数据 |
|:-----|:-----|:-----|
| 第1重 | 过程评价缺失→AI能力不被承认 | 职称只看结果不看AI过程 |
| 第2重 | 能力验证滞后→AI迭代远快于评审周期 | 3-5年vs6-12月 |
| 第3重 | 超级个体无通道→所有政策以单位/团队为申请主体 | 算力券、人才引进均无个人通道 |

效果：逻辑递进，政策制定者一眼看懂问题层次。

### 政策对比表格法（找政策空位）

当提案涉及"现有政策的空白"时，使用横向对比表：

```
| 城市/地区 | 政策名称 | 补贴主体 | 是否覆盖目标人群 |
|:----------|:---------|:---------|:-----------------|
| 北京      | 算力券   | 企业     | ❌               |
| 上海      | 智能算力券| 企业/高校 | ❌               |
| 深圳      | 算力资源券| 中小企业 | ❌               |
| ...       | ...      | ...      | ...              |
```

效果：横向对比→空位一目了然→提案价值定位自然显现。

## 策略模式

### "延伸覆盖"策略（首选用法）

不造新轮子，在现有成熟政策框架上做合理延伸：

```
现有政策 [企业算力券] 已证明框架可行
    ↓
论证：同一框架可延伸覆盖 [个人超级个体]
    ↓
关键差异点：DeepSeek技术的成熟使"个人级补贴"变得经济可行
    ↓
结论：不需要新制度，只需要扩展覆盖范围
```

**优势**：降低政策制定者的认知成本，采纳率显著高于"全新方案"。

### 经费可控论证

政策制定者最关心"要多少钱"。必须给出精确的财政可行性计算：

```
[DeepSeek定价] → 百万token约0.5元
              → 个人月均50元
              → 1,000名用户全年仅6万元
              → 对比企业算力券百万级投入 → 财政可行性一目了然
```

### 个人案例作为活证据

当你自己是这个提案方向的实践者时，务必写入提案：

```
> "通过自主构建的AI科研系统（Synthos），一人完成了35篇学术论文"
```

比任何实验室数据都更有说服力——你就是活证据。这是"专业壁垒"维度的极致体现。

### 分阶段实施路径

提案的"对策建议"必须包含可落地的时间表：

| 阶段 | 时间 | 内容 | 规模 |
|:-----|:-----|:-----|:-----|
| 调研论证 | 1-2月 | 摸底需求、制定标准 | 案头工作 |
| 试点启动 | 3-6月 | 首批发放 | 50-100人 |
| 评估推广 | 6-12月 | 效果评估+推广 | 全省/全市 |
| 制度固化 | 12月+ | 纳入正式人才体系 | 制度化 |

## 全过程自动化管线

### Phase 1 — 选题与数据准备（选题层）

```
选题来源：
  ├── 政策热点 → 发改委/卫健委年度重点
  ├── 科研反哺 → 你已完成论文中的政策含义
  ├── 调研发现 → 临床/工作中的真实痛点
  └── 数据驱动 → 公开数据集分析显示的趋势
```

**数据审计（L0.5门）**：提案中每个数字必须追溯源头

| 数据源类型 | 可信度 | 来源渠道 |
|:-----------|:------:|:---------|
| 公开数据集实验（PIMA/WDBC等） | ⭐⭐⭐ | 代码产出+CSV文件 |
| 卫健委/统计局公开数据 | ⭐⭐⭐ | 官网公报 |
| 知网/万方检索统计 | ⭐⭐ | 可复现检索 |
| 文献引用 | ⭐⭐ | 论文PDF可查 |
| 个人调研/估算 | ⭐ | 标注"估算" |

### Phase 2 — 提案生成（三段式管线）

借鉴paper-pipeline的NotebookLM Q&A逐段生成流程：

```bash
# Step 1: 关联到NotebookLM项目（或新建）
notebooklm use <project_id>

# Step 2: 生成"基本情况"段
notebooklm ask "基于笔记本中的参考文献和数据，撰写提案的'基本情况'部分。
要求：
1. 开篇直入主题，用数据说明问题的严重性
2. 引用相关政策文件（标注来源[N]）
3. 控制在500字以内"

# Step 3: 生成"存在问题"段
notebooklm ask "基于上述分析，列出3-5个核心问题及根因。
要求：
1. 每个问题必须有数据支撑（来自NB source）
2. 根因分析要有逻辑链，不是简单归因
3. 控制在800字以内"

# Step 4: 生成"对策建议"段
notebooklm ask "针对上述问题，提出可操作的对策建议。
要求：
1. 每条建议必须有责任主体（谁来做）
2. 有可行性分析（为什么能做）
3. 有优先级排序
4. 控制在800字以内"
```

### Phase 3 — 论文反哺（科研→提案反向管道）

每篇完成论文的"Policy Implications"或"Discussion"部分，自动提取为提案原料：

```python
# 提取逻辑（概念参考，非实际代码）
从论文中提取：
1. 核心发现 → 转化为"存在问题"的数据支撑
2. 数据趋势 → 转化为"基本情况"的背景叙述
3. 方法创新 → 转化为"对策建议"的技术路线
4. 局限性 → 转化为"需解决的关键瓶颈"
```

**论文→提案映射清单（建议维护在 Notion/Airtable）**：

| 3D虹膜分割 | 基层医疗AI设备准入 | Dice=0.9825, 27K样本 | ✅ 已投稿 |

## 陷阱

1. **🔴 不要在提案中使用英文数据**：提案读者是政策制定者，需用中文呈现，指标加通俗解释（如"AUC=0.843→模型识别准确率84.3%"）。
2. **🔴 建议不可太技术化**：不说"采用3D眼球模型约束的虹膜分割算法"，说"采用先进AI眼动识别技术"。
3. **🔴 数据不可过时**：提案引用政策文件必须为最新版本（1年内）。2023年的文件标注时需确认是否已被取代。
4. **🔴 不可照搬论文数字**：Dice=0.9825在论文中是好结果，在提案中读者无感——转化为"比传统方法提升X%的识别准确率"。
5. **⚠️ 建议必须有执行路径**："建议加强AI监管"太虚→"建议市卫健委牵头制定医疗AI设备临床准入评估指南，参考国家药监局AI医疗器械分类目录"。
6. **🌱 一次多产**：一篇论文至少可以同时产出1篇中文综述+1篇提案。不要浪费论文中的政策含义。
