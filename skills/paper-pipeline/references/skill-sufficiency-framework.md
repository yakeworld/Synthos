# Skill Sufficiency Evaluation Framework

> 评估一个科研系统（如 Synthos）的技能集是否完整，用于 Paper Discussion - Limitations 和 Future Work 的论证支撑。
> 实战验证: Synthos 论文（2026-05-23）

## 核心理念

系统论文的 Limitations 不是"知道有什么问题"，而是**先尝试修复，修复不了才承认局限**。这个框架帮助系统性评估哪些缺口可以填补，哪些必须写入论文。

## 10 维评估框架

| # | 维度 | 评估问题 | 典型缺口信号 |
|:-:|:-----|:---------|:-------------|
| 1 | 文献检索与知识获取 | 覆盖哪些数据库？有无回退机制？ | 单源依赖，无fallback |
| 2 | 知识提取与管理 | 能提取结构化信息吗？支持多模态？ | ⚠️ **纯文本盲区** → 常见缺口 |
| 3 | 关联发现与研究空白 | 有跨文献关系挖掘吗？Gap定位？ | 无知识图谱 |
| 4 | 假设生成与实验设计 | 基于什么哲学？有替代假设吗？ | ⚠️ **实验不执行** → 最常见致命缺口 |
| 5 | 论证表达与论文写作 | IMRaD？图表？LaTeX？ | 无结构化输出 |
| 6 | 观点验证与质量保证 | 反方验证？置信度？幻觉检测？ | 无验证机制 |
| 7 | 项目管理与计划执行 | 长时程流转？子任务调度？ | ⚠️ **单线程** → 常见 |
| 8 | 外部吸收与自我进化 | 能吸收外部知识吗？有进化机制？ | 无自改进 |
| 9 | 跨学科桥接 | 概念对齐？类比推理？ | 领域锁定 |
| 10 | 可复现性与学术诚信 | 数据防伪？过程留痕？ | 无溯源 |

## 评估方法论

### Step 1: 列举所有技能

列出系统的所有原子/模块/技能，分类为：
- **核心认知原子** — 主科研管线的必需组件
- **扩展/元技能** — 辅助能力、进化机制
- **基础设施** — 执行引擎、工具链

### Step 2: 逐维评估

对每个维度，用三个问题判断：

| 问题 | 意义 |
|:-----|:------|
| 覆盖了没有？ | 是否存在对应的原子/技能 |
| 覆盖深度够不够？ | 是否完整覆盖了该维度的所有子能力 |
| 有缺口吗？ | 哪些子能力缺失或薄弱 |

### Step 3: 分类缺口

```
发现缺口
  ├── 可通过现有工具零成本修复 → 修复，不写入论文
  ├── 需要外部项目吸收 → 搜索候选，评估吸收可行性
  │     ├── 可吸收 → 执行吸收，不写入论文
  │     └── 不可吸收（许可证/架构/成本） → 写入 Limitations
  └── 超越当前技术边界 → 写入 Limitations
```

### Step 4: 吸收候选评估（仅当 Step 3 发现可吸收缺口时）

| 维度 | 权重 | 评估方法 |
|:-----|:----:|:---------|
| 互补性 | 0.25 | 是否精准填补功能空白 |
| 集成成本 | 0.25 | 是否零Python/纯curl terminal |
| 许可证 | 0.15 | MIT/Apache优先 |
| 社区活跃 | 0.15 | Stars≥50, 1年内更新 |
| 文档质量 | 0.20 | README完整，有示例 |

## 常见缺口模式（系统论文通用）

### 缺口 A: 实验执行盲区

**症状**: "能策划完美实验但不能运行"  
**常见候选吸收源**:

| 来源 | 集成方式 | 评分 |
|:-----|:---------|:----:|
| **Hermes terminal + Docker**（已有） | SKILL.md声明 `allowed-tools: terminal`，docker run 执行 | **5.0 🔥** |
| E2B REST API | `curl -X POST https://api.e2b.dev/...` | 4.5 |
| Modal serverless | 需API key | 3.0 |

### 缺口 B: 多模态盲区

**症状**: "只能读文字不能读图"  
**常见候选吸收源**:

| 来源 | 集成方式 | 评分 |
|:-----|:---------|:----:|
| **Hermes vision_analyze**（已有） | 直接在EXT SKILL.md中声明 `allowed-tools: vision_analyze` | **4.5 🔥** |
| GROBID REST API | 部署 Java 服务，curl 调 REST | 4.0 |
| PyMuPDF/Marker | 间接 terminal 调用 | 3.5 |

### 缺口 C: 单线程执行

**症状**: "不能并行处理子任务"  
**常见候选吸收源**:

| 来源 | 集成方式 | 评分 |
|:-----|:---------|:----:|
| **Hermes delegate_task**（已有） | task-router 新增并行路由规则 | **5.0 🔥** |
| Hermes max_concurrent_children | 配置参数调整 | 4.0 |

## 最终输出

评估完成后，生成：
1. **可修复的缺口清单** → 立即执行修复
2. **不可修复的缺口清单** → 写入 Paper Discussion - Limitations
3. **每个不可修复缺口的理由** → 供同行评审参考
